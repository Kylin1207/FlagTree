# TLE Distributed API Detailed Design

## 1. Goals

- Provide a hardware-agnostic distributed programming model under `triton.experimental.tle`.
- Keep backend-specific implementation details (cluster launch, DSMEM instructions, collectives) out of user-facing Python API.
- Make the semantics of `mesh`, `sharding`, `remote`, `barrier`, and future `reshard`/`distributed_dot` explicit enough for stable compiler lowering and diagnostics.

## 2. Scope and Non-Goals

### In Scope

- API contracts for:
  - `device_mesh`
  - `S/P/B`, `sharding`, `make_sharded_tensor`
  - `distributed_barrier`
  - `remote`
  - `reshard` (M4 design)
  - `distributed_dot` (M5 design)
- Launch and synchronization model.
- Backend capability and lowering contracts.
- Error model and test strategy.

### Out of Scope

- Full implementation details for non-NVIDIA backends.
- Auto-parallelization or automatic mesh inference from full graphs.
- Cross-host runtime transport implementation details (NCCL/RDMA plugin specifics).

## 3. External Design Inputs

This design aligns with:

- CUDA Thread Block Cluster / DSMEM model (`cluster.sync`, distributed shared memory address mapping).
- PyTorch DTensor `DeviceMesh` + `Placement` (`Shard`, `Replicate`, `Partial`) and `redistribute` behavior.
- JAX mesh/partition model where unspecified mesh axes imply replication.
- OpenXLA/GSPMD sharding representation using per-axis tiling and explicit replication.

References are listed in Section 15.

## 4. Core Terminology

- Mesh axis: named logical axis in `device_mesh` (for example `node_x`, `device`, `cluster_x`, `block`).
- Split axis (`S`): tensor dimension partitioned across mesh axis.
- Partial axis (`P`): local values are partial reductions and need reduce semantics.
- Broadcast axis (`B`): tensor dimension replicated across that mesh axis.
- Sub-mesh: mesh slice derived from `mesh[...]`.
- Shard ID: target coordinate/index in current mesh scope for remote access.

## 5. API Contract Matrix

| API | Semantics | Current Status |
| --- | --- | --- |
| `device_mesh(topology)` | Defines named, shaped logical topology with deterministic coordinate linearization. Supports slicing/reshape/flatten. | Implemented |
| `S`, `P`, `B`, `sharding(...)` | Constructs validated `ShardingSpec`. No communication is generated by annotation itself. | Implemented |
| `make_sharded_tensor(...)` | Binds logical handle + sharding metadata + optional shape. | Implemented |
| `distributed_barrier(mesh=None)` | Synchronization primitive over current distributed scope. | Implemented cluster-wide and static sub-mesh lowering on NVIDIA; irregular/dynamic subgroup patterns still fail fast |
| `remote(tensor_or_buffer, shard_id, scope=None)` | Declares remote access target. Supports pointer and buffered-tensor materialization path. | Implemented |
| `reshard(tensor, spec)` | Changes sharding state and triggers collective/scatter/gather operations. | Not implemented (M4) |
| `distributed_dot(a, b, c=None)` | Distributed GEMM over cluster scope, behavior inferred from sharding. | Not implemented (M5) |

## 6. Detailed Semantics

### 6.1 `device_mesh`

#### Contract

- `topology` is an ordered mapping of hierarchy levels.
- Each level can be:
  - scalar size: `{"device": 4}`
  - list of named dimensions: `{"node": [("x", 2), ("y", 2)]}`
- Axis names must be unique globally.
- Shape dimensions must be positive integers.

#### Linearization and Slicing

- Coordinate-to-linear index is row-major over mesh dimensions.
- `mesh[idx_or_slice]` returns a sub-mesh with:
  - selected physical IDs
  - filtered shape and dimension names
- Empty sub-mesh is illegal.

#### Rationale

- Matches established mesh abstractions in DTensor/JAX/OpenXLA while preserving explicit axis naming.

### 6.2 `S/P/B`, `sharding`, `make_sharded_tensor`

#### Contract

- `split`: list length equals tensor rank when shape is known.
- Each split entry maps one tensor dim to zero-or-more mesh axes.
- `partial`: set of mesh axes in partial-reduction state.
- Broadcast axes are inferred as all mesh axes not in split or partial.
- Split and partial axes are disjoint.

#### State Encoding

- `axis_state(axis)` returns `S`, `P`, or `B`.
- Unspecified mesh axes are always `B`, consistent with JAX/OpenXLA replication semantics.

### 6.3 `distributed_barrier(mesh=None)`

#### Current Contract (M3)

- Without `mesh`: synchronize current cluster scope.
- With `mesh`: infer cluster launch shape and synchronize cluster scope.
- If `mesh` is a strict sub-mesh, current implementation infers subgroup descriptor and emits group metadata when builder support is available.
- If runtime/builder lacks group-aware entrypoint, it fails fast with explicit `NotImplementedError`.
- For `group_kind=submesh` with static `group_mask`, NVIDIA lowering uses a software subgroup barrier:
  - reserve a small shared-memory scratch region (`ttg.shared`) for counter/phase.
  - map scratch pointers to subgroup leader CTA via `mapa.shared::cluster`.
  - perform CTA0 participant atomic arrive/spin/release.
  - keep CTA-level entry/exit barriers so all threads in each CTA observe the same progression.

#### Target Contract (M3 hardening)

- With sub-mesh input, only members of sub-mesh participate.
- Non-members are no-op at semantic level.
- Compiler must ensure non-members do not wait on participants.
- Detailed lowering plan is tracked in `docs/tle/design/distributed_barrier_submesh.md`.

#### Correctness Rule

- All participants in the same synchronization group must execute matching barrier calls; otherwise behavior is undefined and may deadlock.

### 6.4 `remote(tensor_or_buffer, shard_id, scope=None)`

#### Input Types

- Pointer tensor path (`tl.tensor` of pointer type, shared addrspace).
- Buffered tensor path (`tleg.alloc` output), where remote metadata is attached and pointer is materialized later via `tleg.local_ptr`.
- Buffered tensor path is metadata-only until `tleg.local_ptr(...)`: duplicate `remote(buffered_tensor, ...)` annotation is rejected to avoid ambiguous semantics.

#### `shard_id` Contract

- Compile-time: int or coordinate tuple/list (linearized via `scope` mesh).
- Runtime: scalar int32 value/tensor.
- For runtime shard ID, lowering preserves a dedicated carrier marker.
- Buffered tensor path performs early shard-id validation and fails fast for invalid compile-time/range/type usage.

#### Scope and Launch

- If `scope` provided, cluster shape is inferred from mesh.
- Current implementation requires `num_ctas=1` with mesh-driven cluster launch.

#### Memory and Sync Rule

- Producer-then-remote-consumer access requires explicit synchronization (`distributed_barrier`) to ensure visibility and ordering.

### 6.5 `reshard(tensor, spec)` (M4)

#### State Transition Rules

| Source -> Target | Meaning | Communication |
| --- | --- | --- |
| `[ ] -> [S]` | Scatter | Yes |
| `[S] -> [ ]` | Gather | Yes |
| `[P] -> [ ]` | Reduce | Yes |
| `[B] -> [S]` | Local slice | No |
| `[S] -> [B]` | All-gather | Yes |
| `[P] -> [B]` | All-reduce | Yes |
| `[B] -> [P]` | Illegal | Error |

#### M4 Minimum Deliverable

- Implement legality checks and clear diagnostics first.
- Introduce a backend-agnostic collective IR abstraction.
- Lower to backend collectives based on capability.

### 6.6 `distributed_dot(a, b, c=None)` (M5)

#### Target Semantics

- Cluster-scoped matrix multiply where data movement strategy is inferred from sharding.
- `a`/`b` require cluster-level sharding metadata.
- `c` is optional accumulator with compatible distribution.

#### M5 Minimum Deliverable

- One supported pattern first:
  - `a`: split on M over cluster axis.
  - `b`: split on N over cluster axis or replicated.
- Explicit unsupported-pattern diagnostics.

## 7. Launch Model

### 7.1 Current

- Mesh-driven launch inference is used by distributed APIs.
- Cluster dims inferred from mesh axis names in priority:
  - axis names containing `cluster`
  - axis names containing `block`
  - fallback to full mesh product
- `num_ctas=1` indicates one program maps to one block.

### 7.2 Constraints

- Cluster dimensions must be compatible with grid dimensions at launch time.
- If user-specified cluster dims conflict with inferred dims, compilation fails with explicit error.

## 8. Lowering Architecture

### 8.1 Frontend

- Parse and validate distributed metadata in Python API.
- Attach semantic attributes/markers for later lowering.

### 8.2 TLE Dialect (Target Shape)

- `distributed_barrier` op with optional group description.
- `remote` op/pointer marker with compile-time or runtime shard source.
- `reshard` op with source/target sharding specs.
- `distributed_dot` op with inferred communication pattern metadata.

### 8.3 Backend-NVIDIA Mapping

- `group_kind=cluster` barrier lowers to cluster arrive/wait primitive.
- `group_kind=submesh` barrier lowers to software subgroup synchronization over cluster-shared scratch + atomics.
- Remote shared-memory pointer lowers through cluster address mapping path.
- Runtime shard ID lowering consumes dedicated marker and preserves i32 runtime value.
- Future collectives map to backend collective runtime or generated communication kernels.

## 9. Capability Model

Backend capability keys (compile-time query):

- `supports_cluster_launch`
- `supports_dsmem_remote`
- `supports_submesh_barrier`
- `supports_collective_reshard`
- `supports_distributed_dot`

Rules:

- API remains available; unsupported paths fail with actionable diagnostics.
- Fallback strategies (if any) must be explicit and never silently degrade correctness.

## 10. Error Model and Diagnostics

Mandatory diagnostic classes:

- Shape/rank mismatch.
- Unknown mesh axis.
- Split/partial overlap.
- Illegal reshard transition (`B -> P`, unsupported pattern).
- Invalid `shard_id` type/rank/range.
- Unsupported backend capability for requested operation.
- Launch contract violation (`num_ctas`, conflicting cluster dims).

## 11. Test Strategy

### 11.1 Unit Tests

- `device_mesh`: parse/slice/reshape/linearization.
- `sharding`: legality and inferred broadcast axes.
- `remote`: shard ID normalization and type checks.
- `reshard`: legality matrix (M4).

### 11.2 Integration Tests

- Same-cluster remote DSMEM read/write with deterministic producer-consumer ordering.
- Runtime shard ID path with IR marker and backend instruction checks.
- Sub-mesh barrier selective synchronization (M3 hardening).
- Reshard end-to-end collective behavior (M4).
- Distributed dot numeric and distribution correctness (M5).

## 12. Security and Safety Notes

- Remote pointer access is constrained to cluster scope and backend address-space rules.
- Compiler must reject illegal address-space remote usage.
- Synchronization misuse must surface warnings/errors when statically detectable.

## 13. Open Questions

1. Should sub-mesh membership be explicit in API (`group=`) or implicit via slicing only?
2. Should `remote` accept multi-axis coordinate tuples without explicit `scope` when tensor has bound sharding?
3. For unsupported `reshard` patterns, do we fail hard or permit explicit opt-in fallback kernels?
